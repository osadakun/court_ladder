## 1. 目的

本システムは、バドミントンの「勝ち上がり負け落ち」方式において、各コートの対戦順、試合結果、参加者の移動先を本部で一元管理する Web システムである。

本システムにより、次を実現する。

- 試合結果の入力は本部のみで行う
- 結果入力後、勝者 / 敗者の移動先コートと待機順を自動反映する
- 本部モニターに全コートの進行状況を常時表示する
- 同じ内容を公開 URL で配信し、参加者・保護者・関係者が閲覧できる
- 手書き運用による誤記、順番違い、見落としを減らす
- 履歴を保存し、誤入力時の復旧を可能にする

## 3. 用語定義と前提

### 3-1. 大会

本システムが管理する単位。大会ごとにコート数、公開 URL、状態、revision を持つ。

### 3-2. コート

- シングルスコート: `1..singles_court_count`
- ダブルスコート: `(singles_court_count+1)..(singles_court_count+doubles_court_count)`
- 合計コート数 = `singles_court_count + doubles_court_count`
- 設定可能範囲: シングルス 0〜20、ダブルス 0〜20（ただし合計 2〜20）
- シングルスのみ、ダブルスのみの大会も可
- 数字が小さいほど上位コートとする（各種別内で）
- 数字が大きいほど下位コートとする（各種別内で）
- `courts` レコードは大会作成時に合計コート数分作成する
- コートには `court_type`（`singles` / `doubles`）を持たせる

### 3-3. エントリー

試合に参加する単位。従来仕様の「参加単位」に相当する。

- シングルスエントリー: 1エントリー = 1名
- ダブルスエントリー: 1エントリー = 2名
- 1大会にシングルスとダブルスのエントリーが混在可能

### 3-4. メンバー

エントリーを構成する個人。シングルスでは1名、ダブルスでは2名。

### 3-5. ダブルス時のペア

- ダブルスエントリーは2名のメンバーで構成する
- エントリー作成時点で `members[]` はちょうど 2 件必須とする
- 大会状態が `live` の間も、管理者操作によりメンバー組み替えを許可する。組み替え後の変更は将来の試合にのみ影響し、過去の試合記録は変更前のメンバー構成を保持する
- 試合作成時にエントリーのメンバー構成・表示名のスナップショットを保存し、履歴の正確性を担保する

### 3-6. 待機列

各コートで試合待ちしているエントリーの並び。`queue_items.queue_position` により順序を持つ。

### 3-7. 現在対戦

あるコートに現在割り当て済みの対戦カードを指す。初版では「試合開始」操作は設けず、待機列から自動生成された時点でそのコートの現在対戦とみなす。

### 3-8. 試合状態

- `in_progress`: 現在対戦として割り当て済み
- `finished`: 結果が確定した
- `cancelled`: 管理操作により無効化された

`cancelled` は「終了結果のロールバック」と「現在対戦の手動解消」を含み、区別は `cancel_reason` で行う。

### 3-9. 結果種別

- `normal`: 通常終了
- `retired`: 途中棄権
- `walkover`: 不戦勝
- `abandoned`: 打ち切り

`取り消し` は結果種別ではなく試合状態である。

`abandoned` は大会終了時等に進行中の試合を途中経過として記録するための種別。勝者なし（`winner_entry_id = null`）、移動処理なし。スコアは任意で記録可能。

### 3-10. エントリー状態

- `active`: 有効
- `paused`: 休止
- `withdrawn`: 大会離脱

`withdrawn` は大会からの棄権を表し、「途中棄権」とは別概念である。

### 3-11. 大会状態

- `preparing`: 準備中。設定・参加登録・初期配置が可能。結果入力は不可
- `live`: 進行中。通常運用。結果入力、移動、自動生成、公開更新を行う
- `ended`: 終了。読み取り専用。結果入力や通常修正は不可

### 3-12. 大会 revision

`tournaments.revision` は大会単位のグローバル更新番号である。

- 公開画面または管理ダッシュボードの表示内容に影響する更新が成功するたびに 1 加算する
- 例: 結果確定、結果取り消し、手動移動、待機列並び替え、コート停止 / 再開、大会開始 / 終了 / 再開、公開設定変更、エントリー状態変更に伴う待機列自動除外
- `meta.revision`、Realtime 通知、ETag 生成の基準とする
- `revision` が加算された更新は、Supabase Realtime を通じてクライアントへ自動通知される
- クライアントは revision が進んだ場合のみ再取得する

### 3-13. 停止コート

一時的に使用しないコート。停止中コートは次を満たす。

- 新規の現在対戦を自動生成しない
- 自動移動先として選ばれない
- 待機列は保持する
- 停止解除後に通常運用へ戻る

### 3-14. プレビューデータ

CSV 取込プレビューは、**DB 上の一時テーブル** に TTL 付きで保持する。

- サーバー再起動後も commit 可否を判断できること
- マルチインスタンス時も一貫した挙動を取れること
- 初版の TTL は **30分** とする
- preview には `expires_at` と `consumed_at` を持たせる
- 期限切れ preview、または既に消費済み preview は commit 不可とする

### 3-15. 管理者アカウントのスコープ

初版の `admin_accounts` は **単一運営主体向けのシステム共通管理者** とする。

- 大会ごとの権限分離は初版対象外
- 1つの管理者アカウントは全大会へアクセス可能
- 将来、マルチテナント化する場合は `admin_tournament_roles` 等の導入を想定する

### 3-16. 公開 URL / public token

- 大会ごとにユニークな `public_token` を発行する
- `public_token` は推測困難なランダム値とし、URL 安全文字列で管理する
- 公開ページはログイン不要だが、検索エンジンには掲載させない

## 4. 業務ルール

### 4-1. 勝ち上がり負け落ちルール

コート `c` で行われた試合について、勝者・敗者の移動先は次の通りとする。

- 勝者: `c-1` 方向へ 1 つ上がる
- 敗者: `c+1` 方向へ 1 つ下がる

移動は **同一コート種別（singles / doubles）の範囲内** でのみ行う。シングルスコートのエントリーはシングルスコート間でのみ移動し、ダブルスコートのエントリーはダブルスコート間でのみ移動する。

### 4-2. 境界ルール

- 同種別の最上位コートで勝った場合は最上位コートに残留する
- 同種別の最下位コートで負けた場合は最下位コートに残留する
- 例: シングルス7コート（1〜7）、ダブルス3コート（8〜10）の場合
  - シングルス: 1コートで勝ち → 1コート残留、7コートで負け → 7コート残留
  - ダブルス: 8コートで勝ち → 8コート残留、10コートで負け → 10コート残留
  - 7コートで負けても8コートへは移動しない。8コートで勝っても7コートへは移動しない

### 4-3. 停止コートスキップルール

自動移動先のコートが停止中の場合、**同一コート種別の範囲内で**同じ方向へさらに 1 コート先を探索し、最も近い稼働中コートへ移動する。

### 4-4. 稼働中コートが存在しない場合

停止コートをスキップしても同種別・同方向に稼働中コートが存在しない場合、当該エントリーは **現在コートに残留**する。

### 4-5. 待機列の順序

待機列の順序は、各コート内で管理される `queue_position` により決定する。付与ルールは次の通りとする。

1. 初期配置時
   - エントリー作成時（手動追加 / CSV取込）に指定した初期コートの待機列末尾に追加
2. 結果確定時の移動
   - 移動先コートの待機列末尾へ追加
   - 追加順は **結果確定トランザクションの確定順**
3. 手動並び替え
   - 管理者が指定した順に **全件再採番**
4. 現在対戦手動解消時
   - `requeue_mode = tail_keep_order` の場合、`entry_a` → `entry_b` の順で同一コート待機列末尾へ戻す

### 4-6. 現在対戦の自動生成

現在対戦は次の条件をすべて満たす場合に自動生成する。

- 大会状態が `live`
- コート状態が `active`
- 当該コートに `in_progress` 試合が存在しない
- 待機列に有効エントリーが 2 組以上存在する

動作は次の通りとする。

- 待機列の先頭 2 組を現在対戦にする
- 選ばれた 2 組は待機列から除外する
- 試合状態は `in_progress` とする
- 公開画面および管理画面に即時反映する

### 4-6-2. 同一チーム対戦回避

大会設定 `allow_same_team_match` が `false` の場合、待機列への自動挿入時に同一チーム連続を回避する。

回避ロジック（結果確定後の移動先コート待機列への追加時）:

- 移動先コートの待機列の先頭エントリーと、追加されるエントリーが同一チームの場合、先頭の次の位置（2番目）ではなく、**先頭エントリーの直前**に挿入する
- 正確には: 待機列先頭から順に見て、同一チームのエントリーが連続している区間の**直前**に挿入する
- 待機列の全エントリーが同一チームの場合は、通常通り末尾に追加する（対戦回避は不可能なため許容）
- `allow_same_team_match = true` の場合は常に末尾追加（従来動作）
- チーム未所属（`team_id = null`）のエントリーは同一チーム判定の対象外とする

### 4-7. 一意性ルール

同一大会内において、1 エントリーは次を同時に満たしてはならない。

- 複数の待機列に存在する
- 複数の `in_progress` 試合に存在する
- 待機列と現在対戦の両方に同時に存在する

### 4-8. エントリー状態変更ルール

- `active → paused` または `active → withdrawn` の際、当該エントリーが待機列に存在する場合は **待機列から自動除外**する
- この自動除外は `movement_logs` に `movement_reason = auto_remove_due_to_status` として記録する
- 当該エントリーが `in_progress` 試合に含まれている場合、状態変更は **409** で拒否し、先に結果確定または `clear-current-match` を要求する
- `paused` / `withdrawn` から `active` に戻しても、自動では待機列へ戻さない
- `active` に戻ったエントリーは **未所属** 状態とし、管理者が手動で任意コートの待機列へ追加する

### 4-9. 大会状態遷移

大会状態は次の遷移を持つ。

- `preparing → live`: 管理者が「大会開始」を実行
- `live → ended`: 管理者が「大会終了」を実行
- `ended → live`: 管理者が「大会再開」を実行

状態ごとの制御は以下の通りとする。

| 大会状態 | 設定変更 | 参加登録 | 初期配置 | 結果入力 | 手動修正 | 公開閲覧 |
|---|---|---:|---:|---:|---:|---:|
| `preparing` | 可 | 可 | 可 | 不可 | 可 | 可 |
| `live` | 一部可 | 追加のみ可 | 不可 | 可 | 可 | 可 |
| `ended` | 不可 | 不可 | 不可 | 不可 | 原則不可 | 可 |

補足:

- `live → ended` は、大会内に `in_progress` 試合が 1 件も存在しない場合にのみ許可する
- `in_progress` 試合が存在する場合、`POST /actions/end` は **409 ACTIVE_MATCHES_EXIST** を返し、先に結果確定または現在対戦の手動解消を要求する
- `live → ended` および `ended → live` の成功時は `revision` を加算し、公開画面 / 管理画面へ更新通知する
- `ended` 大会は管理者が削除可能（FR-16 参照）

### 4-10. 停止コートの状態変更条件

- 停止操作は管理者のみ可能
- 現在対戦が存在するコートの停止は、初版では **結果確定または手動解消後に限る**
- 停止中コートでは新規の現在対戦を自動生成しない

### 4-11. 表示名ポリシー

公開画面の表示名は **自動生成** とし、手動での上書きは不可とする。

- シングルス: `学年：メンバー名（チーム名）`
- ダブルス: `学年1：メンバー1名・学年2：メンバー2名（チーム名）`
- チーム未所属の場合はチーム名を省略する
- 学年は `grade` フィールドの値をそのまま表示する（例: `中3：田中太郎（田中チーム）`）
- エントリーが所属するチームの **チームカラー** により、エントリーの **背景色** を色分けして表示する（従来の先頭色丸は廃止）
- チームカラーはアプリのプリセットパレット（20色）から自動割り当てされ、管理者が手動変更もできる

## 5. 利用者と権限

### 5-1. 管理者

管理画面にログインして利用するユーザー。初版の権限は単一ロールとし、管理者間の権限差は設けない。

管理者ができること:

- 大会設定
- メンバー / エントリー登録
- CSV 一括登録（初期コート指定含む）
- 選手個別追加（初期コート指定含む）
- 結果入力
- 手動修正
- コート停止 / 再開
- 履歴確認
- 大会開始 / 終了 / 再開
- 公開 URL の有効 / 無効切り替え

### 5-2. 閲覧者

公開 URL または本部モニターを閲覧するユーザー。

閲覧者ができること:

- 現在対戦の確認
- 待機列の確認
- 更新時刻の確認
- 大会名やコート状況の確認

閲覧者ができないこと:

- 結果入力
- 並び替え
- 移動
- 参加者編集
- 履歴修正

## 6. 認証・セッション・同時操作

### 6-1. 認証方式

初版の認証方式は、**Supabase Auth による email / パスワード認証（JWT）** とする。

- OAuth は初版対象外
- 管理者アカウントのセルフ登録は不可
- 初期管理者は Supabase Dashboard または CLI で事前作成する

### 6-2. セッション管理

- ログイン後は Supabase Auth が発行する **JWT（access_token + refresh_token）** で認証状態を保持する
- access_token の有効期限は Supabase のデフォルト（**1時間**）に従う
- refresh_token により自動更新する
- ログアウト機能を提供する
- トークン失効後は再ログインを要求する

### 6-3. セキュリティ前提

- パスワードは Supabase Auth が管理し、平文保存されない
- 管理画面は HTTPS 前提とする
- JWT は Authorization ヘッダで送信する
- API は Edge Functions で JWT を検証し、認可を行う
- 監査ログに主要操作を記録する

### 6-4. 複数管理者の同時操作

初版では複数管理者のログインを許可する。ただし、同じコートや同じ試合に対する同時更新には **楽観的ロック** を適用する。

- 結果入力時は `matches.version` を送信する
- コート停止 / 再開、待機列並び替え、現在対戦解消 / 再計算は `courts.version` を送信する
- エントリー編集 / 状態変更は `entries.version` を送信する
- `version` 不一致の場合は `409 VERSION_CONFLICT` を返し、最新状態の再読込を促す

## 7. システム構成

本システムは Web アプリとして構成する。

- 管理画面: 本部 PC またはタブレットで利用
- 公開画面: 本部モニターに常時表示
- 公開 URL: スマートフォン、タブレット、PC から閲覧可能
- アプリサーバー: 状態管理、認証、履歴保存
- データベース: 大会、待機列、試合、移動履歴、監査ログ、preview を保存

### 7-1. リアルタイム更新方式

公開画面および本部モニターの自動更新は、**Supabase Realtime（WebSocket）** を使用する。

- `tournaments` テーブルの `revision` カラムの変更を Supabase Realtime の **Postgres Changes** で購読する
- 変更検知後、クライアントは 0〜500ms のランダム遅延を入れて snapshot API を再取得する
- `GET /snapshot` には `ETag: W/"rev-<revision>"` を付与する
- `Cache-Control: public, max-age=1, must-revalidate` を返す
- WebSocket 接続が切れた場合は再接続を試み、失敗が続く場合は **3 秒間隔のポーリング** にフォールバックする
- 表示更新の目標は、結果確定から **3 秒以内** とする

### 7-2. 公開 URL の形式

- 大会ごとに **ユニークな公開 URL** を発行する
- URL には推測困難な **ランダムトークン** を含める
- 例: `/public/{token}`

### 7-3. 公開 URL のアクセス制御方針

- 公開画面はログイン不要
- ランダムトークンにより、実質的に限定共有とする
- `noindex, nofollow, noarchive` を設定し、検索エンジンへの露出を抑制する
- サイトマップには掲載しない
- ただし、URL を知っている者はアクセス可能である

## 8. 機能要件

### FR-01. 大会設定

管理者は大会ごとに以下を設定できること。

- 大会名
- 公開ページタイトル
- 開催日
- シングルスコート数（0〜20、初期値 0）
- ダブルスコート数（0〜20、初期値 0）
- ※ シングルス + ダブルス合計は 2〜20
- ゲームポイント（初期値 21）
- 大会状態
- 公開 URL の有効 / 無効
- 公開待機列表示件数（初期値 5、設定範囲 1〜10）
- 同一チーム対戦許容（`allow_same_team_match`、BOOLEAN、初期値 `true`）

制約:

- コート数（シングルス / ダブルス各）は大会開始後に変更不可
- 公開ページタイトルは大会開始後も変更可
- 大会状態の変更操作には確認ダイアログを表示する
- `allow_same_team_match` は大会開始後も変更可

### FR-02. 管理者アカウント管理

初版では最低限、次を満たすこと。

- 管理者ログイン画面を提供する
- 事前作成済みアカウントでログインできる
- ログアウトできる
- 認証失敗時にエラーメッセージを表示する

### FR-03. メンバー / エントリー登録

管理者はエントリーを登録・編集・休止・大会離脱設定できること。

エントリー登録項目:

- 形式（シングルス / ダブルス）
- チーム（team_id, 任意）
- 状態（有効 / 休止 / 大会離脱）
- 初期コート番号（`initial_court_no`、必須。シングルスエントリーはシングルスコート、ダブルスエントリーはダブルスコートの範囲内で指定）
- メンバー配列 `members[]`

メンバー登録項目:

- 氏名（`management_name`、必須）
- 学年（`grade`、必須。固定11選択肢: 年長, 小1, 小2, 小3, 小4, 小5, 小6, 中1, 中2, 中3, 大人）

登録方法:

- CSV 一括登録（FR-04 参照）
- 管理者画面からの個別追加

ルール:

- シングルスでは `members[]` は 1 件必須
- ダブルスでは `members[]` は 2 件必須
- 表示名はメンバー名・学年・チーム名から自動生成される（§4-11 参照）
- 休止 / 大会離脱中のエントリーは自動対戦生成対象から除外する
- エントリー編集 / 状態変更には `entries.version` による楽観的ロックを適用する
- エントリー作成時、指定した `initial_court_no` のコート待機列末尾に自動追加する

状態変更時の動作:

- `active → paused` または `active → withdrawn` の際、当該エントリーが待機列に存在する場合は **待機列から自動除外**する
- この自動除外は `movement_logs` に `movement_reason = auto_remove_due_to_status` として記録する
- 当該エントリーが `in_progress` 試合に含まれている場合、状態変更は **409** で拒否し、先に結果確定または `clear-current-match` を要求する
- `paused` / `withdrawn` から `active` に戻しても、自動では待機列へ戻さない
- `active` に戻ったエントリーは **未所属** 状態とし、管理者が手動で任意コートの待機列へ追加する
- 待機列自動除外により公開盤面が変化した場合は `revision` を加算する（Realtime 経由で自動通知）

### FR-04. CSV 一括登録

管理者は CSV によりメンバー / エントリーを一括登録できること。

初版で対応する列:

- 形式（シングルス / ダブルス）
- チーム名
- メンバー1氏名
- メンバー1学年（固定11選択肢）
- メンバー2氏名（ダブルス時のみ）
- メンバー2学年（ダブルス時のみ）
- 初期コート番号（必須）

動作:

- 取込前にプレビューを表示する
- 不正行はエラー理由を表示する
- 正常行のみ取り込むか、全件取り消すかを選択できる
- preview は 30 分で失効する
- 初期コート番号のバリデーション: シングルスエントリーはシングルスコート範囲、ダブルスエントリーはダブルスコート範囲内であること

### FR-05. 初期配置

全エントリーはCSV取込時または手動追加時に初期コート（`initial_court_no`）を必須で指定する。エントリー作成と同時に指定コートの待機列末尾に追加される。

従来の自動配置（allocation preview / apply）フローは廃止する。

管理者は大会開始前に、待機列の手動並び替え（FR-06）により初期順序を調整できる。

### FR-06. コート別待機列管理

システムは各コートごとに待機列を保持できること。

必須要件:

- 各コートに 1 つの待機列を持つ
- 順番を明示的に表示する
- 管理者は手動で順番変更できる
- 管理者は任意のエントリーを別コートへ移動できる
- 管理者は **未所属エントリーを任意コート待機列へ追加**できる
- エントリー状態が `paused` または `withdrawn` の場合、現在対戦に含めない

### FR-07. 現在対戦の自動生成

システムは各コートについて、業務ルール 4-6 の条件で現在対戦を自動生成すること。

### FR-08. 試合結果入力

管理者は本部画面から各コートの現在対戦に対して結果を入力できること。

入力フロー（1画面で完結）:

1. 結果種別を選択する（通常終了 / 途中棄権 / 不戦勝 / 打ち切り）
2. 種別に応じた入力欄を表示する:
   - **通常終了**: 勝者選択（左 / 右）→ 敗者得点（0〜game_point-1）
   - **途中棄権 / 不戦勝**: 勝者選択（左 / 右）のみ
   - **打ち切り**: entry_a の得点と entry_b の得点をそれぞれ入力（勝者選択なし）
3. 通常終了時、スコアは自動的に「勝者: game_point、敗者: 選択値」となる
4. 例: game_point=21 で勝者が左の場合 `21-16`、右の場合 `16-21`

入力項目:

- コート番号
- 対戦カード
- 結果種別（通常終了 / 途中棄権 / 不戦勝 / 打ち切り）
- 勝者選択（左 / 右）（打ち切り時は不要）
- 敗者得点（通常終了時、0〜game_point-1）
- entry_a 得点 / entry_b 得点（打ち切り時のみ）
- 備考（任意）

判定ルール:

- 通常終了時は、勝者選択と敗者得点からスコアを自動算出する
- 途中棄権（`retired`）、不戦勝（`walkover`）の場合はスコア入力不要
- 打ち切り（`abandoned`）の場合は勝者なし（`winner_entry_id = null`）、移動処理なし。スコアは入力値をそのまま記録

制約:

- 結果入力は管理者のみ
- 大会状態が `live` のときのみ入力可
- 同一試合に対する二重確定を防止する
- 確定前に確認ダイアログを表示する
- 入力日時、入力者、入力時の version を履歴に残す

### FR-09. 勝者 / 敗者の移動処理

結果入力後、移動先コートを計算して **プレビューダイアログ** を表示する。

- プレビューには「勝者 → コートX 待機列へ」「敗者 → コートY 待機列へ」を含む
- 管理者が確認して「確定」を押した場合のみ、実際の移動処理を実行する
- 「キャンセル」した場合は結果入力画面に戻る

確定後、システムは自動で以下を行うこと。

1. 勝者の移動先コートを計算する
2. 敗者の移動先コートを計算する
3. 停止コートをスキップして最終移動先を決定する
4. それぞれ移動先コートの待機列末尾へ追加する
5. 元コートの試合状態を `finished` にする
6. 影響のあるコートで次対戦を自動生成する
7. 公開画面を更新する
8. 移動履歴を保存する
9. `tournaments.revision` を加算する（Realtime 経由で自動通知）

打ち切り（`abandoned`）の場合:

- 勝者・敗者の移動処理は行わない
- 両エントリーは元コートの待機列末尾へ `entry_a` → `entry_b` の順で戻す
- 試合状態は `finished` にする
- 移動履歴は `movement_reason = abandoned_requeue` で記録する

### FR-10. 結果取り消しとロールバック

管理者は、終了済み試合に対して「結果取り消し」を実行できること。ただし、自動ロールバックが成立する条件を満たす場合に限る。

自動ロールバック成立条件:

- 対象試合が `finished` 状態である
- 勝者・敗者が移動済みである
- 勝者・敗者が移動先コートで **まだ別試合に割り当てられていない**
- 勝者・敗者が移動先待機列から削除可能である

自動ロールバック動作:

1. 勝者・敗者を移動先待機列から削除する
2. 元コートの待機列末尾へ、元の対戦順（A → B）で戻す
3. 対象試合の状態を `cancelled`、`cancel_reason = rollback` に変更する
4. 影響のあるコートの現在対戦を再計算する
5. 監査ログと移動履歴にロールバック記録を残す

不成立時:

- 自動ロールバックは行わず **409 AUTO_ROLLBACK_NOT_ALLOWED** を返す
- 管理者は手動修正モードを用いて復旧する

### FR-11. 手動修正

管理者は運営上の例外に対応するため、以下を行えること。

- エントリーの手動移動
- 未所属エントリーの待機列への追加
- 待機順の入れ替え
- 現在対戦の手動解消
- コート停止 / 再開
- エントリー状態の変更
- 大会再開
- 結果取り消し可能性の確認
- 現在対戦の再計算（手動トリガー）

制約:

- 重要操作は確認ダイアログを必須とする
- 手動修正後は affected コートに対して現在対戦の再計算を行う
- 手動修正の前後状態は監査ログに残す

現在対戦の手動解消 (`clear-current-match`):

- 初版の `requeue_mode` は `tail_keep_order` のみ対応とする
- `tail_keep_order` では、現在対戦の `entry_a`、`entry_b` を **この順**で同一コート待機列の末尾へ戻す
- 対象試合は `state = cancelled`、`cancel_reason = manual_clear` とする
- この操作は `movement_logs` に `movement_reason = manual_requeue` として記録する
- 実行後、当該コートは `recalculate` と同じルールで次試合生成可否を再判定する

現在対戦の再計算 (`recalculate`):

- `recalculate` は **現在対戦が存在しない稼働中コート**に対して、待機列変更後の次試合自動生成を手動トリガーする操作である
- 当該コートに `in_progress` 試合がある場合は **409 CURRENT_MATCH_EXISTS** で拒否する
- 当該コートが停止中の場合も **409 COURT_STOPPED** で拒否する
- 条件を満たし、かつ有効エントリーが 2 組以上いる場合のみ現在対戦を自動生成する

### FR-12. コート停止 / 再開

管理者はコートを停止 / 再開できること。

停止時の動作:

- 停止中コートでは現在対戦を新規作成しない
- 停止中コートは自動移動先から除外する
- 停止中コートの待機列は保持する
- 必要に応じて管理者が待機列を他コートへ手動移動できる

再開時の動作:

- 待機列に 2 組以上いて現在対戦がなければ、自動生成対象に戻す
- 再開により盤面が変わる場合は `revision` を加算する（Realtime 経由で自動通知）

### FR-13. 公開表示

公開 URL では、以下の内容を閲覧できること。

- 大会名
- 公開ページタイトル（設定時）
- 更新時刻
- 大会状態
- 1〜N コートの一覧
- 各コートの状態（稼働中 / 停止中）
- 各コートの現在対戦
- 各コートの待機順
- 各エントリーのチームカラー
- 各コートの待機人数

表示方針:

- 本部モニターで見やすい一覧レイアウトにする
- スマホでも閲覧可能なレスポンシブ表示にする
- 手動更新なしで自動反映する
- 待機列は上位 `3〜5件` を標準表示とし、既定値は `5件`
- 残り件数がある場合は「他 X 件」と表示する
- 公開画面のタイトル表示は、`public_page_title` が設定されていればそれを優先し、未設定時は `name` を用いる
- 公開 API は内部 ID を返さない

### FR-14. 本部モニター表示

本部モニター向け画面は公開画面をベースとし、次を満たすこと。

- 1 画面で全コートが見渡せる
- 文字サイズが大きく、遠くからでも判読しやすい
- 自動更新される
- 必要に応じてフルスクリーン表示できる

### FR-15. 履歴保存・出力

システムは以下を履歴として保存すること。

- 試合履歴
- 結果種別
- スコア
- 移動履歴
- ロールバック履歴
- コート停止 / 再開履歴
- 管理者操作履歴

出力:

- CSV 形式でエクスポート可能
- 大会終了後、全結果を **PDF** でエクスポートできる（初版は CSV を最低限、PDF は可能であれば対応）
- エクスポート内容: 全試合結果（コート、対戦カード、スコア、勝敗）、エントリー別成績サマリー
- 大会終了後も参照可能

### FR-16. 大会データ削除

管理者は大会終了後、大会データを完全に削除できること。

- 削除対象: 大会に紐づく全データ（エントリー、チーム、試合、待機列、移動履歴、監査ログ、preview）
- 削除前に確認ダイアログを表示する
- 削除前にエクスポート（PDF / CSV）を促すガイダンスを表示する
- 削除は `ended` 状態の大会に対してのみ実行可能とする
- 削除は取り消し不可能な操作である

## 11. データ整合性要件

システムは次の整合性を保証すること。

1. 同一大会内で、1 エントリーは同時に複数の待機列へ存在しない
2. 同一大会内で、1 エントリーは同時に複数の `in_progress` 試合へ存在しない
3. 同一大会内で、1 コートに同時に複数の `in_progress` 試合は存在しない
4. `finished` または `cancelled` 済み試合に対して通常の結果再入力はできない
5. `ended` 大会では更新操作を受け付けない
6. 停止中コートは自動移動先および自動対戦生成対象から除外される
7. `PATCH /queue` は送信された `entry_ids[]` が処理時点の待機列構成と完全一致しなければならない
8. ロールバックや手動移動では `queue_items` の一意制約を守る処理順序を保証する

## 12. 非機能要件

### 12-1. 性能

- 結果確定から公開画面反映まで **3 秒以内** を目標とする
- 同時閲覧数 100 名程度で閲覧に支障がないこと
- 管理操作は通常 3 秒以内に応答すること

### 12-2. 可用性

- 公開画面は通信断後に自動再接続すること
- WebSocket 切断時は再接続を試みること
- フォールバックとしてポーリングが動作すること

### 12-3. セキュリティ

- 管理画面はログイン必須
- 公開画面はログイン不要
- 監査ログを保存する
- パスワード平文保存を禁止する

### 12-4. 端末対応

- 管理画面: PC、タブレット
- 公開画面: PC、タブレット、スマートフォン
- モダンブラウザ対応

### 12-5. プライバシー

- 公開画面の表示名はメンバー名とチーム名から自動生成される
- 未成年利用を想定し、検索エンジンへの露出を抑える

## 13. 受入基準

次を満たした場合、初版仕様を満たしたとみなす。

1. 管理者がログインし、管理画面へアクセスできる
2. 大会状態が `preparing` の間は結果入力できない
3. 大会開始後に結果入力できる
4. 任意コートで結果を入力すると、勝者は上位方向、敗者は下位方向へ自動移動する
5. 移動先コートが停止中ならスキップして次の稼働中コートへ移動する
6. 稼働中コートが存在しない場合は現在コートに残留する
7. 移動先コートでは待機列末尾に追加される
8. 待機列先頭 2 組から現在対戦が自動生成される
9. 公開 URL で全コートの現在対戦と待機順上位 N 件が見られる
10. 公開画面は手動更新なしで自動反映される
11. 同じ試合結果を二重確定できない
12. 複数管理者が同じコートを同時操作した場合、後勝ちではなく排他制御で片方が拒否される
13. 結果取り消し可能条件を満たす場合に限り、自動ロールバックできる
14. 結果取り消し可能条件を満たさない場合、システムが取り消しを拒否して手動修正へ誘導する
15. `in_progress` 試合が残っている状態では大会終了できず、409 で拒否される
16. `paused` / `withdrawn` に変更した待機列中エントリーは自動除外され、公開画面へ反映される
17. `paused` / `withdrawn` から `active` に戻したエントリーは自動復帰せず、手動追加操作で待機列へ戻せる
18. 試合履歴、移動履歴、監査ログが保存される
19. CSV 一括登録と初期自動配置が行える
20. ダブルスエントリーで 2 名 1 組のエントリー登録ができる

## 14. 初版方針として確定した事項

| 項目 | 初版方針 |
|---|---|
| シングルス / ダブルス | エントリー単位で混在可能。コートはシングルス/ダブルスに分離 |
| コート構成 | singles_court_count + doubles_court_count（合計2〜20） |
| スコア入力 | game_point 制（勝者選択 → 敗者得点選択） |
| スコア形式 | 勝者は game_point、敗者は 0〜game_point-1 |
| 打ち切り | 勝者なし、移動なし、スコア任意記録 |
| 公開表示 | 学年：名前（チーム名）+ チームカラー背景色 |
| URL 公開範囲 | インターネット公開 + ランダムトークン + noindex |
| 待機列表示数 | 既定値 5 件、設定で変更可 |
| 参加者検索 | 初版対象外 |
| CSV 一括登録 | 初版対応（初期コート・学年列を含む） |
| 選手個別追加 | 初版対応（管理者画面から） |
| 初期配置 | 自動配置廃止。エントリー作成時に初期コート必須 |
| 同一チーム対戦回避 | スイッチで切替可能（allow_same_team_match） |
| 連戦制御 | 初版対象外 |

## 16. 実装上の補足指針

- 「現在対戦」は別テーブルを持たず、`matches.state = in_progress` により表現する
- `courts.current_match_id` は高速参照用の冗長列として持ってよい
- `courts.current_match_id` の更新は、試合自動生成 / 結果確定 / 結果取り消し / 現在対戦手動解消 / コート再開後の再計算と **同一トランザクション内** で行う
- 結果確定、移動、待機列更新、次試合生成は同一トランザクションで処理する
- ロールバックは自動成立条件を厳密に判定し、中途半端な巻き戻しを禁止する
- `queue_items` の一意制約を守るため、ロールバックや手動移動では「移動先から削除 → 移動元 / 追加先へ挿入」の順序を守る
- UI 上の確認ダイアログ対象は少なくとも次を含む
  - 結果確定
  - 結果取り消し
  - コート停止
  - 大会開始
  - 大会終了
  - 大会再開
  - 大量移動 / 一括並び替え
  - 大会データ削除
